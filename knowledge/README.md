# Miloco 知识库

> 小米智能家居 AI 管理平台 Miloco 2.0 的知识库，产品视角与研发视角合一。
>
> **当前状态**：本页描述的目录结构已完成迁移落地；02-strategy / 04-testing 内容由各团队持续补充。

---

## 一、这个知识库是什么

帮 AI / 新人快速搞懂 Miloco：每个模块解决什么问题、怎么设计的、代码在哪。

它**不是**源码的自然语言副本。凡是代码、`--help`、OpenAPI、`settings.schema.json`、git 历史里已是单一事实源的信息，这里只指路，不复制。

## 二、抽象规范（写每一篇都先对照）

### 总标尺

> 读完一篇，读者能在脑子里画出模块关系图、知道去哪个文件看代码、明白关键设计为什么这么定；**但不需要背任何一个数字、不需要复述任何一段函数内部流程**。

### 三个粒度档

| 档位          | 写什么                                                                                | 出现在                                                     |
| ------------- | ------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| **L1 概念级** | 是什么、解决什么问题、能力与边界、模块职责                                            | overview、strategy、features 产品面、external-deps         |
| **L2 架构级** | 分层、数据流（框图级）、核心类名 + 文件路径、关键设计决策的「为什么」、对外契约的语义 | overview、features 研发面、external-deps                   |
| **L3 实现级** | 函数内部步骤、阈值数字、字段 schema、方法签名、版本纠偏流水账                         | **知识库一律不写**，留给代码注释 / schema / `--help` / git |

判断任何一句话该不该留：先看它属于哪一档，落到 L3 就删除或改成指路。

> **例外：06-dev-guide、07-design 不走三档，以各章「颗粒度」说明为准（06-dev-guide 是操作步骤级，07-design 是设计规范条目级，两者天然超出三档边界）；其余目录（包括看似偏研发的 05-external-deps——external-deps 只记集成约束和责任边界，SDK 自身 API 细节交给 SDK 官方文档）一律走三档。**

### 保留 / 删除对照

左列是该留的（L1 / L2 及其派生的配置指路、术语），右列是该删的（L3）。不走三档的 06-dev-guide / 07-design 见例外说明，留删以各章「颗粒度」为准。

| 保留                                                                 | 删除                                                         |
| -------------------------------------------------------------------- | ------------------------------------------------------------ |
| 模块职责、解决什么问题                                               | 函数内部 0~N 步流程                                          |
| 分层 / 数据流框图                                                    | 每层耗时、GC 触发条件、状态机逐态转移细节                    |
| 核心类名 + 文件路径                                                  | 类的每个方法签名                                             |
| 关键设计决策的「为什么」（如为何用 DeepSORT 而非纯 IoU）             | 选定后的每个参数取值（`max_age_sec=3.0` 这类）               |
| 配置「在哪配 / 在哪看」                                              | 配置项逐个默认值清单（同一配置项：位置与语义留，具体取值删） |
| 对外契约的语义（如某状态字段有哪几种含义，可带必要锚点常量）[^1]     | 字段的类型 / 长度 / 格式 schema、渲染格式字符串              |
| 稳定的术语与概念（tier_a / tier_c / tier_u、陌生人池，定义见术语表） | `某版默认` / `数据类默认 X 但 YAML 覆盖 Y` 等版本纠偏        |

[^1]: 锚点常量的边界见「边界速判 · 状态锚点常量」。

### 跨篇统一约定

- **指路而非复制**：涉及配置 / 端点 / 命令，统一写「在哪配、在哪看」，保留位置与语义、删掉具体取值。例：「感知参数集中在 `settings.yaml::perception` 段，字段定义见 `settings.py` 的 `PerceptionSettings`」。
- **代码位置**：核心类后标文件路径，如 `IdentityEngine（perception/engine/identity/engine.py）`，让读者一跳到代码，替代逐个列方法。
- **设计决策**：保留「为什么这么选」，删除「选了之后每个参数是多少」。
- **版本表述**：只写当前稳定形态，不写演进流水账；「曾经怎样、某版改过」交给 git。

### 边界速判

几类卡在 L2/L3 之间的中间态，统一这样判：

- **结构性计数**（如 18 张表、四层流水线）：属框图级，留；但只写「有几张 / 几层」，不展开「第 N 张表叫什么、含哪些字段」。
- **当前技术栈**：组件名留，具体版本号删（版本交给依赖声明与 git）。
- **状态锚点常量**：契约语义可带必要的锚点（如 `503=引擎未就绪`、枚举取值名），但不展开该状态的完整字段 schema 或全部错误码表。
- **自研封装层**（Proxy / Adapter / Client）：封装类的设计意图与对外契约写 features；被封装的外部 SDK 本身的能力边界 / 版本兼容 / 已知限制写 external-deps，两篇互相指路。

综合以上规范，一段合格的 features 研发面写法如下：

> **感知流水线**采用 `MultimodalCollector → Gate → Identity → Omni` 四层。Gate 做变化门控，无变化直接跳过下游省算力；Identity 用 DeepSORT 跟踪并关联身份（为何不用纯 IoU：静止人物易丢 track）；Omni 调用 VLM 出场景描述与规则匹配。核心编排在 `PerceptionEngine（perception/engine/api.py）`，各层参数集中在 `settings.yaml::perception` 段。

它给了分层、数据流、一个设计决策的「为什么」、核心类 + 路径、配置指路——没有一个阈值数字、没有一段函数内部流程。

## 三、整体结构

```
knowledge/
├── README.md            # 本页：唯一入口 = 定位 + 抽象规范 + 整体结构 + 全局索引 + 术语表
├── 01-overview/         # 产研一体 · 全局概览（项目地图）
├── 02-strategy/         # 产品 · 定位 / 洞察 / 竞品 / 规划
├── 03-features/         # 产研一体 · 每个核心模块一篇（知识库主体）
├── 04-testing/          # 产研一体 · 评测方法论
├── 05-external-deps/    # 研发 · 外部 SDK / 协议 / 服务依赖与责任边界
├── 06-dev-guide/        # 研发 · 上手与运维（环境 / 安装 / 启动 / 故障排查）
└── 07-design/           # 设计 · 设计规范
```

各目录具体文件详见「四、全局索引」。

### 各部分存什么 / 不存什么 / 边界

**01-overview** — 全局概览

- 存：项目是什么、整体架构一张图、四条主链路（设备控制 / 感知 / 规则 / Agent）的端到端数据流框图、技术栈一览、认证模型这类全局事实。
- 不存：任何单模块内部设计、阈值、方法清单。
- 边界：overview 是地图（模块之间怎么串），features 是街景（单模块内部），互不重复。

**02-strategy** — 产品战略

- 存：产品定位、目标用户与洞察、竞品对比、阶段规划——这些是「判断」。
- 不存：实现、架构、代码。
- 边界：overview 讲「是什么的事实」，strategy 讲「为什么做这市场、和谁比、往哪走的判断」。

**03-features** — 核心模块（知识库主体）

- 每个核心模块一篇，篇内统一两段：
  - 产品面（L1）：背景与目标、功能能力、典型场景、能力边界。
  - 研发面（L2）：架构设计（分层 / 数据流框图）、核心类名 + 文件路径、关键设计决策的「为什么」、对外契约的语义。
- 存：模块职责、设计意图、核心抽象、去哪看代码。
- 不存：函数内部流程、阈值清单、REST/CLI 端点全表、字段级 schema、版本纠偏。
- 边界：本模块讲透即可，全局串联交给 overview；自研逻辑进这里，外部 SDK 归 external-deps。

**04-testing** — 评测方法论

- 存：评测框架总览、各能力评测（模型 / 身份 / Agent / 端到端）方法、评测集评选方案、评分逻辑、准出标准、自动化评测平台。
- 不存：某次评测的原始数据与跑分明细。
- 边界：跨模块的「怎么验收」，与 features 的「怎么设计」分开。
- 颗粒度：方法论级，不写单次跑分明细。

**05-external-deps** — 外部依赖

- 存：每个外部 SDK / 协议 / 服务一篇——它是谁的、我们怎么用、集成约束、版本兼容性、已知限制、责任边界。
- 不存：自研逻辑的实现细节。
- 边界：外部维护的 SDK / 协议 / 服务进此目录，自研逻辑进 features。

**06-dev-guide** — 上手与运维

- 存：环境准备、安装、启动、常见开发工作流、故障排查（症状 → 排查 → 解决）。
- 不存：模块设计原理（去 features）、配置默认值清单（指向 schema）。
- 边界：features 讲「系统怎么设计」，dev-guide 讲「人怎么跑起来、出问题怎么查」。
- 颗粒度：可照着做的操作步骤级，不深入原理。

**07-design** — 设计规范

- 存：统一的设计语言、视觉规范、组件与布局模式，保证各端各模块界面体验一致。
- 不存：具体业务功能逻辑（归 features）。
- 边界：只管「界面长什么样、怎么搭」，功能逻辑归 features。
- 颗粒度：可复用的规范条目级，不写页面功能逻辑实现。

## 五、术语表

> 重构时统一收敛到此处，features 内不再各自重复定义。

| 术语                                | 含义                                                                                                                                                           |
| ----------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 感知流水线                          | 设备感知的核心处理管道，采用 `MultimodalCollector → Gate → Identity → Omni` 四层架构，核心编排在 `PerceptionEngine`                                            |
| Omni                                | 感知流水线中负责场景理解的编排层，调用 VLM（MiMo）输出场景描述与规则匹配                                                                                       |
| VLM                                 | 视觉语言模型，能同时理解图像与文本的多模态大模型；配置入口在 `model.omni`（见 [开发指南「配置分段与用途」表](06-dev-guide/dev-guide.md#配置分段与用途)）       |
| Agent                               | OpenClaw 上运行的智能体，通过 Skill 调用 Miloco 能力                                                                                                           |
| OpenClaw                            | Miloco 所基于的 Agent 框架与插件平台                                                                                                                           |
| MiOT                                | 小米 IoT SDK，用于发现与控制米家设备                                                                                                                           |
| tier_a / tier_c / tier_u            | 身份样本三层：用户登记样本 / 系统沉淀样本 / 陌生人未确认池（即陌生人池，见下行）                                                                               |
| 陌生人池                            | 未识别身形的暂存与聚类区，供主动注册抽样（对应 tier_u）                                                                                                        |
| Skill                               | OpenClaw 插件提供给 Agent 的能力单元（`miloco-*` 前缀）                                                                                                        |
| STATIC / DYNAMIC 规则               | 静态规则直接执行设备动作；动态规则通过插件回调让 Agent 决策                                                                                                    |
| task / task_record                  | 任务是持久意图主体（生命周期 + 挂载规则 / 定时 / 记录）；任务记录是其行为统计载体（做了多少 / 多久 / 几次），定义见 [任务管理](03-features/task-management.md) |
| kind（progress / duration / event） | 任务记录的三种统计形态：进度型（目标累积）/ 时长型（计时累积）/ 事件型（次数留痕）                                                                             |
| rollover                            | 周期型任务记录跨周期时归档旧成绩、开新一期的自动滚动机制                                                                                                       |
| 有价值事件（meaningful_events）     | 规则命中 / 语音指令 / 建议三类事件的持久化沉淀，附视频片段，供 Agent 与用户回溯                                                                                |
